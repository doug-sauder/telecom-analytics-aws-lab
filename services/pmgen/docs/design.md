# PMGen Service Design

## 1. Purpose

The pmgen service generates synthetic telecom performance-management events
and publishes those events into the analytics event pipeline.

The service primarily publishes events to a Kafka event broker. It also
maintains a legacy HTTP sender path for local compatibility with the earlier
direct-ingest workflow.

## 2. Responsibilities

The pmgen service performs the following actions:

  - Generates synthetic PM events for a configurable fleet of cells.
  - Emits events at a configurable interval.
  - Publishes events to the configured Kafka topic.
  - Supports a legacy HTTP event sender for direct ingest service posting.
  - Exposes Prometheus producer metrics.
  - Provides CLI commands for continuous production, one-off generation, and
    runtime configuration inspection.

In the overall architecture, pmgen owns simulation behavior and producer-side
delivery attempts.

## 3. Operational Modes

The pmgen service is operated through CLI commands that select how generated
events are used.

| Command | Mode | Behavior |
| --- | --- | --- |
| `pmgen produce` | Kafka producer | Continuously generates PM events and publishes them through the Kafka sender. |
| `pmgen run` | HTTP sender | Continuously generates PM events and sends them through the legacy HTTP sender. |
| `pmgen generate-once` | One-off generation | Generates a single PM event and writes it to stdout. |
| `pmgen config show` | Configuration inspection | Prints the effective runtime configuration. |

The continuous modes (Kafka producer, HTTP sender) use the same generator and
runtime loop. They differ only in the transport selected for event delivery.

The one-off generation mode is intended for payload inspection, schema examples,
and simple local checks. It does not send the generated event.

Configuration inspection is provided as an operator aid so the effective
runtime settings can be checked without starting the generator loop.

## 4. Runtime Design

The service is a Python application, which imports the following packages for
specific functionality:

| Package | Funtionality |
| --- | --- |
| Typer | Command-line interface |
| Pydantic | Configuration and payload models |
| aiokafka | Publishing Kafka events |
| httpx | HTTP event sending |
| prometheus-client | Exposing metrics |


At startup in the default container command, the process:

  1. Loads runtime configuration from `PMGEN_*` environment variables.
  2. Starts the HTTP server to support Prometheus metrics scapes.
  3. Creates a runtime loop with a Kafka event sender.
  4. Generates one PM event per loop iteration.
  5. Sends each event to the configured Kafka topic.
  6. Sleeps for the configured interval before generating the next event.

The Prometheus metrics server runs in the same process as the generator loop.
Metrics are exposed while the continuous `produce` or `run` command is active.

## 5. Interfaces

### 5.1 Kafka Output

The primary output is a Kafka topic containing JSON-encoded PM event messages.

The Kafka sender uses aiokafka `send_and_wait`, sets `acks` to `all`, and uses
the event `entity_id` as the message key. The value is serialized as UTF-8 JSON.

Kafka output is controlled by these environment variables:

| Variable | Default | Description |
| --- | --- | --- |
| `PMGEN_KAFKA_BROKER` | `localhost:9092` | Kafka bootstrap broker used by the producer. |
| `PMGEN_KAFKA_TOPIC` | `pm.events` | Kafka topic where generated events are published. |

### 5.2 HTTP Output

The service exposes a legacy sender path through the `pmgen run` command. That
command continuously posts generated events to the configured HTTP endpoint.

HTTP output is controlled by these environment variables:

| Variable | Default | Description |
| --- | --- | --- |
| `PMGEN_TARGET_URL` | `http://localhost:3000/v1/events` | HTTP endpoint that receives generated events. |
| `PMGEN_REQUEST_TIMEOUT_SECONDS` | `5.0` | Request timeout used by the HTTP client. |

This path is retained for compatibility with the earlier HTTP-based ingestion
flow. The Kafka producer path is the primary event path.

### 5.3 Metrics

The continuous runtime commands expose:

> GET /metrics

The metrics endpoint is served by the Prometheus Python client on the configured
port.

Metrics output is controlled by this environment variable:

| Variable | Default | Description |
| --- | --- | --- |
| `PMGEN_PROMETHEUS_PORT` | `8000` | TCP port for the Prometheus metrics HTTP server. |

pmgen does not currently expose dedicated `/healthz` or `/readyz` endpoints.
Prometheus scrape success is the available process-level signal.

## 6. Event Contract

pmgen emits PM events with the following logical fields:

| Name | Description | Format | Example |
| --- | --- | --- | --- |
| event_id | Event identifier generated by pmgen. | UUID string | 123e4567-e89b-12d3-a456-426614174000 |
| schema_version | Event schema version. | integer | 1 |
| source | Event source identifier. | string | "pmgen" |
| event_time | Event timestamp generated at event creation time. | ISO 8601 timestamp | 2026-05-12T12:34:56Z |
| entity_type | Telecom entity type. | string literal | "cell" |
| entity_id | Telecom cell identifier. | string | "CELL-000123" |
| metrics | PM metric values payload. | object | (see the JSON below) |

The following is an example of a generated PM event as a JSON object:

```json
{
  "event_id": "uuid",
  "schema_version": 1,
  "source": "pmgen",
  "event_time": "2025-12-29T20:45:12Z",
  "entity_type": "cell",
  "entity_id": "CELL-000123",
  "metrics": {
    "dl_prb_util_pct": 73.2,
    "ul_prb_util_pct": 41.8,
    "rrc_conn_avg": 18,
    "drop_rate_pct": 0.7
  }
}
```

Pydantic models enforce the generated event shape. Unknown event fields are
rejected by the model so generator output stays aligned with the ingest
contract.

## 7. Generation Design

Each generated event represents one synthetic cell sample.

For each event, pmgen:

  1. Selects a random cell index from `1` through `PMGEN_CELL_COUNT`.
  2. Formats the cell identifier as `CELL-000001`, `CELL-000002`, and so on.
  3. Generates metric values using the configured scenario.
  4. Validates the metrics and event payload with Pydantic models.
  5. Increments the generated-events Prometheus counter.
  6. Sends the event through the selected transport.

Supported scenarios:

| Scenario | Behavior |
| --- | --- |
| `steady` | Produces bounded random utilization and drop-rate values around a stable per-cell baseline. |
| `busy-hour` | Adds a time-varying sinusoidal demand wave with per-cell offsets to mimic a load spike. |

Unknown scenario names currently fall back to the `steady` behavior.

## 8. Kafka Producer Design

The Kafka producer is created lazily on the first send attempt and reused across
later sends.

Producer settings:

  - `bootstrap_servers` from `PMGEN_KAFKA_BROKER`.
  - JSON value serialization.
  - UTF-8 string key serialization.
  - `acks="all"`.
  - `linger_ms=5`.

For each event send, the service:

  1. Starts the producer if it is not already running.
  2. Serializes the PM event to JSON.
  3. Increments the in-progress send gauge.
  4. Calls `send_and_wait` for the configured topic.
  5. Records send duration when the wait exits.
  6. Decrements the in-progress send gauge.
  7. Increments the success counter if the send completed successfully.
  8. Increments failure counters if the send raised an exception.

The runtime loop sends one event at a time. It does not batch events or run
multiple concurrent producer tasks.

## 9. Delivery Semantics

The intended producer-side behavior is acknowledged Kafka delivery for each
event that `pmgen produce` reports as sent.

The Kafka sender waits for broker acknowledgement before counting an event as
sent. If producer startup, serialization, broker communication, or
acknowledgement fails, the event is counted as failed by the Kafka metrics and
the runtime loop logs the exception.

Failed sends are not retried by pmgen. After a failed send, the runtime loop
continues and generates a new event on the next interval.

Once Kafka accepts an event, downstream delivery, validation, idempotency, and
persistence are owned by Redpanda and ingest. Exactly-once semantics are out of
scope.

## 10. Failure Behavior

### 10.1 Kafka Unavailable

When Kafka is unavailable:

  - The first send attempt fails when producer startup or send acknowledgement
    cannot complete.
  - The failure is logged with the event id and entity id.
  - Kafka failure counters are incremented.
  - The runtime loop continues after the configured interval.

### 10.2 HTTP Target Unavailable

When the legacy HTTP sender target is unavailable:

  - The HTTP request raises an exception or non-success status.
  - The failure is logged with the event id and entity id.
  - Runtime failure stats are incremented.
  - The runtime loop continues after the configured interval.

The custom Kafka delivery metrics do not count HTTP sender failures.

### 10.3 Invalid Runtime Configuration

Configuration is validated by Pydantic at startup. Invalid values such as
non-positive intervals, non-positive cell counts, or invalid Prometheus ports
prevent the command from starting.

### 10.4 Invalid Generated Payload

Generated metrics and events are validated before send. The current generation
functions produce bounded values that satisfy the model constraints. If a future
scenario generates invalid values, validation raises before the event is sent.

### 10.5 Shutdown

On `SIGINT` or `SIGTERM`, the service:

  1. Stops the runtime loop.
  2. Closes the active sender.
  3. Stops the Kafka producer or HTTP client as applicable.
  4. Exits the process.

## 11. Configuration

The service is configured through environment variables with the `PMGEN_`
prefix.

Generation:

  - `PMGEN_INTERVAL_SECONDS`
  - `PMGEN_CELL_COUNT`
  - `PMGEN_SOURCE`
  - `PMGEN_SCHEMA_VERSION`
  - `PMGEN_SCENARIO`

Kafka producer:

  - `PMGEN_KAFKA_BROKER`
  - `PMGEN_KAFKA_TOPIC`

Legacy HTTP sender:

  - `PMGEN_TARGET_URL`
  - `PMGEN_REQUEST_TIMEOUT_SECONDS`

Metrics:

  - `PMGEN_PROMETHEUS_PORT`

Defaults support local development, but Compose should set explicit values for
the integrated stack.

## 12. Metrics

The service uses `prometheus-client` and exposes default Python process metrics
plus custom producer metrics. See [PMGen Metrics](metrics.md) for the full
metrics description.

## 13. Testing

Current repository coverage for pmgen is mostly integration-oriented through
the Compose stack and smoke-test workflow.

Known test gaps:

  - There are no dedicated pmgen unit tests for event generation scenarios.
  - There are no dedicated pmgen unit tests for Kafka sender success and failure
    metric behavior.
  - There are no dedicated tests for signal-driven shutdown cleanup.

## 14. Known Limitations

  - Failed Kafka sends are counted and logged but not retried.
  - The runtime loop sends one event at a time and does not support batching.
  - Unknown scenario names fall back to `steady` instead of failing
    configuration validation.
  - pmgen does not expose dedicated health or readiness endpoints.
  - The HTTP sender path is retained for compatibility but is not the primary
    event path.
  - Random generation is not seeded or externally reproducible.
